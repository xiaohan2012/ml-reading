# Training Table

## Description

A Training Table is the abstraction introduced in [Relational Deep Learning](relational-deep-learning.md) for specifying any predictive task over a relational database. Formally, each row in the training table is a triple:

$$v = (\mathcal{K}_v, t_v, y_v)$$

where $\mathcal{K}_v$ is a set of foreign keys identifying the entity (or entity pair for link prediction) to predict, $t_v$ is the **seed time** controlling which historical data the model may access, and $y_v$ is the ground truth label.

**Two critical properties distinguish training tables from tabular ML training sets**:
1. The training table contains **no input features** ($x_v$). All input signal comes from the relational database via graph message passing — the model must learn to aggregate it.
2. Labels are **computed automatically from historical data** (via time-conditioned SQL queries) without any external annotation, by sliding a time window backward from the current time.

This formulation unifies all three relational task types:
- **Node-level** (entity classification/regression): `(EntityID, seed_time, label)` — 1 foreign key
- **Link-level** (recommendation): `(SourceEntityID, TargetEntityID, seed_time, label)` — 2 foreign keys
- **Higher-order**: `>2` foreign keys (e.g., multi-entity prediction)

The seed time $t_v$ plays a dual role: it defines the model *output* (the entity/time at which to predict) and controls the model *input* (only entities with $\tau(u) \leq t_v$ can be used for message passing), cleanly preventing temporal leakage without any special-casing.
